> Shared behaviour reference for the three "digs/builds a structure over time, from a resource bucket" buildings: [Builder's Hut](Builder's%20Hut.md), [Mine](Mine.md), and [Quarry](Quarry.md). Quarry is the odd one out and is covered in more detail on its own page — **it has no dedicated Java building class at all**, unlike every other building documented so far; it's assembled entirely from a `DefaultBuildingInstance` plus one module. Worth knowing before searching for a `BuildingQuarry.java` that doesn't exist.

### Core shared class
[`AbstractBuildingStructureBuilder`](../../../minecolonies/src/main/java/com/minecolonies/core/colony/buildings/AbstractBuildingStructureBuilder.java) is the base both [`BuildingBuilder`](../../../minecolonies/src/main/java/com/minecolonies/core/colony/buildings/workerbuildings/BuildingBuilder.java) and [`BuildingMiner`](../../../minecolonies/src/main/java/com/minecolonies/core/colony/buildings/workerbuildings/BuildingMiner.java) extend (Quarry doesn't — it doesn't build full structures the same way, see its own doc). It provides:
- **The resource "bucket" system** (`BuildingResourcesModule`, `BUILDING_RESOURCES` in `BuildingModules`): the total material list for a structure is split into staged batches (`BuilderBucket`s) rather than requested all at once — `checkOrRequestBucket()`/`getNextBucket()`/`nextStage()` step through them. This is why a Builder doesn't demand an entire building's worth of cobblestone up front.
- **Work order claiming**: `setWorkOrder()`/`hasWorkOrder()`/`getWorkOrder()` track a single claimed `IBuilderWorkOrder` by ID (re-resolved from the colony's work manager each time, not cached) — cancelling a work order (`onWorkOrderCancellation()`) resets in-progress requests, needed resources, and progress position for every assigned citizen.
- **Progress tracking**: `progressPos`/`progressStage`/`progressCounter` record where in the structure the worker last acted, persisted to NBT but only marked dirty every 50 actions (`COUNT_TO_STORE_POS`) to avoid excessive saves.
- **`getResourceBatchMultiplier()`** — defaults to 1, overridden by the Miner to request 4× normal when no Quarrier is currently assigned to the same building (see the Mine doc) — i.e. a plain Miner without a Quarrier sibling stocks up further ahead.
- Reserved inventory (`keepX`, both Builder and Miner): a full tool set (pickaxe/shovel/axe/hoe, +shears for Builder) is always held back, plus whatever's still needed for the in-progress structure (`getRequiredItemsAndAmount()` pulls straight from the current `BuilderBucket`).

### The Miner ↔ Quarrier ↔ Quarry relationship
This is the most non-obvious system in this group:
1. A **Quarrier** is hired at a **Mine** building, via the `QUARRIER_WORK` module (`MinerBuildingModule`, registered on `BuildingMiner` alongside the ordinary `MINER_WORK` module) — the Mine building hosts both job types, and which one an assigned citizen has determines their behavior.
2. [`JobQuarrier.findQuarry()`](../../../minecolonies/src/main/java/com/minecolonies/core/colony/jobs/JobQuarrier.java) scans every building in the colony whose registry name contains `"quarry"` for one whose `QuarryModule` already lists this citizen as assigned. Until a Quarry claims them, this returns `null`.
3. [`QuarryModule.onColonyTick()`](../../../minecolonies/src/main/java/com/minecolonies/core/colony/buildings/modules/QuarryModule.java) actively looks for exactly this situation: any colony citizen whose job is `JobQuarrier`, not yet assigned to *this* quarry, and whose `findQuarry()` is still null — and claims them. **This is the mechanism behind "the Quarrier must be hired by a mine before it can be hired by the Quarry"**: hiring only ever happens at the Mine building's job-assignment module; the Quarry building can only adopt an already-existing, not-yet-placed Quarrier, never hire a fresh one itself.
4. `JobQuarrier.assignTo()` special-cases `QuarryModule` assignment to not disturb the citizen's actual job "home" — the Mine building remains where the Quarrier is formally employed even after a Quarry has claimed them for actual dig duty. `getWorkStations()` returns the found Quarry building, which is what `IJobWithExternalWorkStations` uses to let the worker operate away from their home building.
5. If a Quarrier is assigned but never gets claimed by any Quarry (none built yet, or all full), they simply have no quarry to work — in that situation, **no shaft mine gets dug at that Mine building either**. Confirmed directly in `EntityAIQuarrier.startWorkingAtOwnBuilding()`: if `job.findQuarry()` returns null, the AI just reports "no quarry assigned" and goes idle — it never falls back to shaft-digging behavior, because `EntityAIQuarrier` is a completely separate AI class from the regular Miner's `EntityAIStructureMiner` and contains no shaft-digging code at all. A citizen with the Quarrier job runs exclusively on quarry-pit logic, whether or not a quarry has actually claimed them yet.

### Extending
Because Quarry has no dedicated class, adding a new quarry-like "dig a pit of a given size" building means writing a new `QuarryModule`-style module (parameterized by height/footprint) and registering it on a `DefaultBuildingInstance`, rather than subclassing anything. A brand new *structure-based* building (a real Builder/Miner-style worker who constructs something incrementally) should extend `AbstractBuildingStructureBuilder` to get the bucket/work-order/progress machinery for free.

### A special case worth flagging if it recurs
The [Builder's Hut](Builder's%20Hut.md) overrides `canBeBuiltByBuilder()` to allow building to exactly **one level above its own current level** — a deliberate one-off bootstrap exception (you can't require a level-5 Builder's Hut to build a level-5 Builder's Hut) rather than the general "a builder needs a hut at least the target level" rule every other building follows. The Builder's Hut is believed to be the *only* building with an override like this — worth explicitly checking for and flagging here if a similar self-referential level-gating override turns up on any future building.
